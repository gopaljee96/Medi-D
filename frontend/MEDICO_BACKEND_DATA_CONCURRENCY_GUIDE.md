## 2. Runtime and persistence configuration

## 2.1 Database profiles
Medico has two data profiles:

1. PostgreSQL profile (`application.properties`)
- JDBC URL: `jdbc:postgresql://localhost:5432/mydb`
- `spring.jpa.hibernate.ddl-auto=validate`
- Meaning: Hibernate validates that DB tables/columns already match entity mappings; it does not auto-create/update schema.

2. H2 profile (`application-h2.properties`)
- In-memory H2
- `spring.jpa.hibernate.ddl-auto=create-drop`
- Meaning: schema is created at startup and dropped at shutdown; mainly for local testing.

## 2.2 Schema migration state
`src/main/resources/schema.sql` currently contains idempotent `ALTER TABLE app_users ... ADD COLUMN IF NOT EXISTS` statements for:
- `active`
- `created_at`
- `updated_at`

Important note:
- In Java entity `User`, these fields are currently marked `@Transient`.
- So they are not part of active JPA persistence mapping at runtime.
- The physical DB may still contain these columns.

Operational implication:
- Current application logic should not depend on persisted values for these transient fields.

## 3. Entity model and effective table design

Below is the effective relational model as represented by current JPA entities.

## 3.1 `app_users` (entity: `User`)
Purpose: authentication identity and role authorization.

Mapped fields:
- `id` (PK, identity)
- `username` (unique, not null)
- `email` (unique, not null)
- `password` (BCrypt hash, not null)
- `role` (enum stored as string)
- `full_name` (nullable)

Transient/non-persisted in current entity mapping:
- `active`
- `createdAt`
- `updatedAt`

Behavior notes:
- login flow checks `user.getActive()`; because field is transient and defaults to true, legacy inactive-state persistence is currently not strongly enforced by mapping.
- uniqueness constraints on username and email are critical and already modeled.

## 3.2 `doctor` (entity: `Doctor`)
Purpose: doctor roster used by scheduling and doctor profile resolution.

Fields:
- `id` (PK)
- `name`
- `specialization`

Behavior notes:
- linkage to user account is name-based (`username` / fallback `fullName`) and not foreign-key based.
- this is a functional design but not the strongest referential model for long-term scale.

## 3.3 `patient` (entity: `Patient`)
Purpose: patient demographics + cumulative medical history.

Fields:
- `id` (PK)
- `name` (required)
- `email`
- `phone`
- `address`
- `medical_history` (TEXT)
- `created_at`

Behavior notes:
- `medicalHistory` works as a timeline text blob; doctor adds timestamped notes.
- legacy accessor alias exists: `getHistoryBlob()` / `setHistoryBlob()`.

## 3.4 `appointment` (entity: `Appointment`)
Purpose: scheduling table for booked slots, blocked slots, cancellations.

Fields:
- `id` (PK)
- `doctor_id` (stored as scalar `Long doctorId`, not FK to doctor table in entity)
- `patient_id` (FK via `@ManyToOne` to patient)
- `appointment_time`
- `status`

Observed statuses in code:
- `BOOKED`
- `BLOCK`
- `CANCELLED`

Design note:
- Because `doctorId` is scalar and not JPA relationship, referential integrity to `doctor` depends on application logic, not FK constraints from current entity mapping.

## 3.5 `medicine` (entity: `Medicine`)
Purpose: inventory used by prescription and pharmacy dispense process.

Fields:
- `id` (PK)
- `name`
- `stock` (critical mutable field)
- `expiry`

## 3.6 `prescription` (entity: `Prescription`)
Purpose: doctor-issued medication order to be dispensed by pharmacist.

Fields:
- `id` (PK)
- `patient_id` (FK via `@ManyToOne`)
- `status` (`PENDING` / `DISPENSED`)
- `created_at`

## 3.7 `prescription_medicines` (generated by `@ElementCollection`)
Purpose: normalized map of medicines and quantities per prescription.

Mapping characteristics:
- join column: `prescription_id`
- map key column: `medicine_id`
- value column: `quantity`

This table represents a many-items composition under a prescription.

## 4. Relationship model

## 4.1 Cardinalities
- Patient 1 -> N Appointment
- Patient 1 -> N Prescription
- Prescription 1 -> N PrescriptionMedicineRows (`prescription_medicines`)

## 4.2 Practical relationship diagram (logical)

`User(role=DOCTOR)`
-> (name-based matching logic)
`Doctor`
-> referenced by `Appointment.doctorId`

`Patient`
-> `Appointment.patient_id`

`Patient`
-> `Prescription.patient_id`

`Prescription`
-> `prescription_medicines` (medicine_id -> quantity)

`Medicine`
-> referenced during stock checks and deductions while dispensing

## 4.3 Referential integrity strength
Strongly modeled:
- patient to appointment (FK through JPA relation)
- patient to prescription (FK through JPA relation)
- prescription to collection rows (`prescription_medicines`)

Weaker/model-by-convention:
- user-doctor linkage (name matching)
- appointment-doctor linkage (`doctorId` scalar)

## 5. Transaction management in Medico

Medico uses both declarative and programmatic transactions.

## 5.1 Declarative transactions (`@Transactional`)
Used in:
- `PharmacyService.doDispenseTransactional(...)`
- multiple methods in `PatientDoctorService`

Behavior:
- transaction begins at method entry via Spring AOP proxy,
- commit on success,
- rollback on exception (`rollbackFor = Exception.class` where configured).

## 5.2 Programmatic transactions (`TransactionHelper`)
`TransactionHelper` wraps `PlatformTransactionManager` manually.

Default settings in helper:
- propagation: `PROPAGATION_REQUIRED`
- isolation: `ISOLATION_READ_COMMITTED`

Exposed patterns:
- `executeInTransaction(...)`
- `executeInTransactionVoid(...)`
- custom isolation overload
- read-only transaction execution

Why this matters:
- gives explicit control over begin/commit/rollback boundaries,
- useful for demos and advanced conditional flows,
- more verbose and developer-responsibility-heavy than declarative style.

## 5.3 Read-only transactions
Helper supports read-only transaction mode for fetch-heavy operations.
This is mainly a hint for optimization and correctness semantics.

## 6. Locking implementation (pessimistic lock)

## 6.1 Where locks are declared
Repository methods using JPA pessimistic write lock:

1. `PrescriptionRepository.findByIdWithLock(Long id)`
- `@Lock(LockModeType.PESSIMISTIC_WRITE)`

2. `MedicineRepository.findByIdWithLock(Long id)`
- declared with pessimistic write lock as well

## 6.2 Where lock is actually used in active flow
In current pharmacy service logic:
- prescription row is fetched with `findByIdWithLock`.
- medicine rows are fetched currently with standard `findById` (not the lock method).

Impact:
- strong lock on prescription record itself,
- medicine row lock is not currently enforced via the lock repository method in active code path.

## 6.3 Why prescription lock still helps
It serializes operations on the same prescription id:
- prevents two dispense operations from marking the same prescription simultaneously without coordination.

But for cross-prescription stock contention on same medicine:
- row-level medicine lock usage is currently weaker than it could be.

## 7. Concurrency behavior by business flow

## 7.1 Pharmacy dispense flow (most critical)
Flow summary:
1. fetch prescription with pessimistic lock,
2. reject if already `DISPENSED`,
3. iterate medicines and validate stock,
4. deduct stock and save medicine rows,
5. set prescription `DISPENSED` and save,
6. commit transaction.

Failure behavior:
- if any medicine has insufficient stock -> throw exception -> rollback whole transaction.
- no partial stock deduction should persist after rollback.

Retry behavior:
- outer wrapper catches lock failures (`PessimisticLockingFailureException`),
- retries up to 3 attempts with incremental sleep.

This is a good baseline for safe dispensing.

## 7.2 Appointment booking flow
Current booking conflict checks are application-level scans (`appointmentRepo.findAll()` then stream checks):
- doctor slot conflict (BOOKED/BLOCK)
- same patient same slot conflict

Concurrency risk:
- two concurrent booking requests can pass check before either inserts, causing race condition if DB has no corresponding unique constraint.

Current guarantee level:
- strong for single-user or low contention traffic,
- not fully race-proof under high parallel requests.

## 7.3 Doctor block-slot flow
Block slot also uses application-level conflict checks with scan pattern.
Same race risk category as booking if highly concurrent.

## 7.4 Doctor profile/self-heal flow
When doctor logs in and record is missing:
- backend may create doctor row on demand.
- if many concurrent requests hit first-time login, duplicate names are possible unless DB uniqueness exists on doctor name.

## 8. Isolation and consistency analysis

## 8.1 Effective isolation level
Programmatic helper defaults to `READ_COMMITTED`.
For declarative methods, DB default/isolation behavior applies unless overridden.

At `READ_COMMITTED`:
- dirty reads are prevented,
- non-repeatable reads and phantoms may still occur depending on query patterns.

Given current patterns:
- pharmacy critical path is mainly protected by explicit lock + transaction.
- appointment scheduling relies more on check-before-insert logic and would benefit from DB constraints.

## 8.2 Atomicity coverage
Strong atomicity in dispense:
- all medicine updates + prescription status update in one transaction.

Moderate atomicity in scheduling:
- each insert/update itself is atomic,
- but pre-check + insert race window remains without DB-level uniqueness enforcement.

## 9. Database hardening recommendations (important)

To move from "functionally working" to "concurrency-robust":

1. Add doctor roster uniqueness
- unique index on `doctor(name)` (or better, introduce `user_id` FK and unique there).

2. Strengthen appointment constraints
- unique index on `(doctor_id, appointment_time)` for active statuses.
- optional unique index for `(patient_id, appointment_time)` for booked appointments.

3. Improve status modeling
- replace free-string statuses with DB-safe enum/check constraints.

4. Use locked medicine fetch in dispense loop
- use `medicineRepo.findByIdWithLock(...)` for row-level stock lock during deduction.

5. Replace `findAll()` conflict checks with targeted repository queries
- better performance and lower race window.

6. Add optimistic locking (`@Version`) where useful
- especially for medicine stock updates if shifting strategy from pessimistic to optimistic.

7. Introduce explicit migration tool
- Flyway or Liquibase for repeatable production-safe schema evolution.

## 10. Current known strengths
- Role-based API protection is implemented.
- JWT auth pipeline is present and consistent.
- Critical dispensing flow has transactional rollback semantics.
- Pessimistic lock and retry mechanism exists for prescription-level contention.
- Admin sync mechanism addresses doctor linkage drift.

## 11. Current known limitations
- Appointment conflict prevention is app-level, not fully DB-enforced.
- User-doctor relationship uses name-matching instead of FK.
- `User.active` currently transient in entity, which weakens persisted account-state semantics.
- Medicine lock method exists but not used in active dispense loop.

## 12. Study summary (for quick revision)
If someone studies only one page, these are the key points:

1. Medico schema centers around users, doctors, patients, appointments, prescriptions, medicines.
2. Prescriptions are normalized with `prescription_medicines` map table.
3. Transactions are used in two styles: `@Transactional` and manual `TransactionHelper`.
4. Pessimistic lock is actively used on prescription in dispensing.
5. Dispensing is atomic and rollback-safe.
6. Booking conflicts are logically checked but still need DB constraint hardening for high concurrency.
7. Doctor linkage currently relies on name alignment plus sync/self-heal mechanisms.
